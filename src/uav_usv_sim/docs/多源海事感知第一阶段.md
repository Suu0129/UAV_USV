# 多源海事感知第一阶段

本阶段建立后续“无人机语义感知 + MID-360 + AIS + 航迹融合 + COLREGs”的公共基础，不改变现有 `landing_boat` 速度控制方式。

## 1. 新增内容

### 统一接口包

`uav_usv_interfaces` 定义：

- `AisContact`、`AisContactArray`：AIS 原始/模拟报文。
- `TrackedObject`、`TrackedObjectArray`：相机、雷达、AIS 和融合节点共用的目标航迹。
- `CollisionRisk`：后续 DCPA/TCPA 风险评估输出。
- `ColregsDecision`：后续 COLREGs 决策输出。

`TrackedObject.source_mask`：

- `1`：LiDAR
- `2`：Camera
- `4`：AIS
- `8`：Fused

第一阶段 AIS 模拟器发布的目标为 `source_mask=4`。后续视觉和 MID-360 节点只需输出相同消息类型。

## 2. TF 结构

标准测试场景发布：

```text
map
├── odom
│   └── landing_boat/base_link
│       ├── landing_boat/hull/front_lidar
│       ├── landing_boat/mid360_link
│       ├── landing_boat/front_camera_link
│       ├── landing_boat/imu_link
│       └── landing_boat/landing_pad_link
└── target_vessel/base_link
```

当前 `map -> odom` 是单位变换，`odom -> base_link` 使用 Gazebo 真值。后续接入 SLAM 后，应由定位系统接管 `map -> odom`，里程计接管 `odom -> base_link`。

`mid360_link` 目前只是预留安装坐标，尚未加入三维雷达传感器。

## 3. 动态合作目标船

模型：

```text
models/target_vessel/
```

特点：

- 7.0 m × 2.6 m 的中型合作船。
- 使用原有 Gazebo `VelocityControl`。
- 命令话题 `/model/target_vessel/cmd_vel`。
- 使用 `BoatWaveFollower` 随浪摇摆。
- 带真实碰撞体，可供后续 MID-360 扫描。

本船仍使用：

```text
/model/simple_boat/cmd_vel
```

没有更换控制插件，也没有改成推进器模型。

## 4. AIS 模拟器

节点：

```text
ais_simulator
```

输入：

```text
Gazebo /world/default/pose/info
```

输出：

```text
/maritime/ais/contacts
/maritime/tracks/ais
/maritime/ais/markers
```

模拟内容：

- MMSI
- 船名
- WGS84 经纬度
- SOG 对地航速
- COG 对地航向
- Heading 船首向
- 船体尺寸
- 位置精度
- 更新周期
- 网络延迟
- 随机丢包
- 位置、速度和航向噪声

AIS 的 COG/Heading 使用海事惯例：从真北开始顺时针。统一航迹的 `Pose` 使用 ROS `map` ENU 坐标。

## 5. 标准场景

配置位于：

```text
config/colregs_scenarios.yaml
```

### 对遇

```bash
ros2 launch uav_usv_sim colregs_test_scenario.launch.py \
  scenario:=head_on
```

两船沿同一直线相向航行，用于 Rule 14。

### 右舷交叉

```bash
ros2 launch uav_usv_sim colregs_test_scenario.launch.py \
  scenario:=crossing_starboard
```

目标从本船右舷方向接近，用于 Rule 15。

### 追越

```bash
ros2 launch uav_usv_sim colregs_test_scenario.launch.py \
  scenario:=overtaking
```

本船从后方接近较慢目标，用于 Rule 13。

### 手动控制本船

标准场景默认自动保持本船速度和航向。关闭自动本船控制：

```bash
ros2 launch uav_usv_sim colregs_test_scenario.launch.py \
  scenario:=head_on \
  auto_ownship:=false
```

此时可以使用原有键盘、RViz 目标控制或后续 Nav2 控制本船。不要同时启动多个本船速度发布者。

## 7. 下一阶段接口约定

MID-360 检测节点输出：

```text
/maritime/tracks/lidar
uav_usv_interfaces/msg/TrackedObjectArray
source_mask=SOURCE_LIDAR
```

无人机视觉节点输出：

```text
/maritime/tracks/camera
uav_usv_interfaces/msg/TrackedObjectArray
source_mask=SOURCE_CAMERA
```

航迹融合节点输入三路航迹，输出：

```text
/maritime/tracks/fused
source_mask 包含实际参与融合的来源，并设置 SOURCE_FUSED
```

风险评估节点输出 `CollisionRisk`，COLREGs 状态机输出 `ColregsDecision`。

## 8. 当前限制

- 目标船和本船仍是速度级 Gazebo 控制，不是推进器动力学。
- AIS 只有一个目标，后续可扩展目标注册表。
- 目标船使用航向 P 控制保持固定航线。
- TF 使用 Gazebo 真值，不是 SLAM/融合定位。
- MID-360 和 IMU 帧已经预留，但三维传感器尚未接入。
- 标准场景目前用于感知、跟踪和规则算法开发，不会自动执行 COLREGs 避让。
